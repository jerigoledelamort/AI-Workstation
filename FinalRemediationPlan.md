## 1. Пересмотр проблем — что изменилось

### 1.1. Ложноположительные проблемы

#### C2 → ИСКЛЮЧИТЬ (ложноположительная)

**Оригинал:** "Ollama не зарегистрирован как служба — Critical"

**Верификация:** Ollama на Windows спроектирован как пользовательское
приложение (tray app), а не как служба. Windows-службы запускаются
под аккаунтом SYSTEM, который в WDDM-режиме (Windows Display Driver Model)
**не имеет доступа к GPU**. Принудительная регистрация Ollama как службы:
- может сломать GPU-инференс (RTX 5070 недоступен для SYSTEM);
- противоречит архитектуре Ollama для Windows;
- создаёт больше риска, чем пользы.

**Факт:** Процесс ollama (PID 7068) + ollama app (PID 17860) работают.
API отвечает на 127.0.0.1:11434. Модели загружаются в VRAM.

**Реальная проблема:** Ollama нет в автозапуске (HKCU Run, Startup folder,
Task Scheduler — не найдено). После перезагрузки потребуется ручной запуск.

**Решение:** Добавить Ollama в автозапуск пользователя (HKCU Run или
Startup folder), НЕ как службу. Это — опциональное удобство, не Critical.

**Категория:** B (опционально)

---

#### M12 → ИСКЛЮЧИТЬ (ложноположительная)

**Оригинал:** "Docker Desktop service — Manual, не запустится автоматически"

**Верификация:** Docker Desktop **ЕСТЬ** в `HKCU:\...\Run`:
`C:\Program Files\Docker\Docker\Docker Desktop.exe`

Docker Desktop автоматически запускается при входе пользователя.
StartType=Manual для com.docker.service — **нормальное поведение**.
Docker Desktop сам управляет жизненным циклом службы при запуске.

**Факт:** Служба Stopped сейчас — потому что Docker Desktop был
остановлен пользователем (или ещё не запущен после перезагрузки).

**Категория:** Исключить. Не является проблемой.

---

#### H3 → ПОНИЗИТЬ (завышенная критичность)

**Оригинал:** "Firewall DefaultInboundAction = NotConfigured — High"

**Верификация:** Registry-значения `DefaultInboundAction` пустые.
В Windows это означает использование встроенного значения по умолчанию.
Для Windows Firewall inbound default = **Block**.

**Факт:** Система уже блокирует все входящие подключения по умолчанию.
NotConfigured — нормальное состояние Windows после чистой установки.
Microsoft не меняет default inbound action через обновления.

**Риск если оставить:** Близок к нулю. Default = Block уже работает.

**Категория:** C (можно оставить). Явная установка Block —
косметическая мера для читаемости конфигурации.

---

#### H1 → ИСКЛЮЧИТЬ (не относится к проекту)

**Оригинал:** "PyTorch cu118 vs CUDA 12.8 — High"

**Верификация:** PyTorch не используется AI Workstation.
Inference полностью через Ollama (собственный runtime, не PyTorch).
PyTorch установлен пользователем для других задач.

**Риск изменения:** Переустановка PyTorch с cu128 может сломать
существующий код пользователя. Несовместимость версий, изменение ABI.

**Категория:** D (исключить). Не наше дело.

### 1.2. Завышенная критичность

#### H5 → ПОНИЗИТЬ с A до B

**Оригинал:** "Две версии Python (3.10 + 3.14) — High, обязательно"

**Пересмотр:** Наличие нескольких версий Python — норма для dev-машины.
Python 3.14.2 установлен Python Software Foundation (легитимно).
С uv (TDR-003) версия системного Python становится нерелевантной —
uv создаёт изолированные окружения с любой версией.

**Реальная проблема:** Не "две версии", а "нет управления версиями".
Решается установкой uv, а не удалением Python 3.14.

**Удаление Python 3.14:** Риск > польза. Пользователь установил его
для своих задач. Удаление может сломать стороннее ПО.

**Категория:** B. Решение — uv, не удаление.

---

#### M7 → ПОНИЗИТЬ с A до B

**Оригинал:** "Стуб Microsoft Store python в PATH — обязательно"

**Пересмотр:** С uv все скрипты будут запускаться в venv.
System Python (и его стуб) не вызывается напрямую.
Стуб раздражает, но не блокирует работу с uv.

**Категория:** B. Опциональное удобство.

### 1.3. Изменения метода исправления

#### H7 — Git autocrlf: изменить метод

**Оригинал:** `git config --global core.autocrlf true`

**Риск оригинального метода:** Глобальный `core.autocrlf true`
конвертирует LF→CRLF при checkout и CRLF→LF при commit.
Это может:
- повредить бинарные файлы;
- сломать shell-скрипты в DevContainers (Linux);
- создать diff-шум;
- вызвать конфликты в кроссплатформенных проектах.

**Новый метод:** `.gitattributes` per-project:
```
* text=auto eol=lf
*.ps1 text eol=crlf
*.bat text eol=crlf
*.cmd text eol=crlf
```

Это гарантирует правильные line endings в репозитории
независимо от глобальных настроек Git.

**Категория:** A (сохраняется), но метод изменён.
