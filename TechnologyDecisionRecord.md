## TDR-001: Inference Engine

### Контекст
Локальный inference LLM на RTX 5070 (12 ГБ VRAM, compute 12.0).
Требуется: загрузка GGUF-моделей, GPU/CPU offload, REST API,
поддержка tool calling, vision, embeddings. Эксклюзивно локально.

### Варианты

| # | Вариант | Плюсы | Минусы |
|---|---------|-------|--------|
| A | **Ollama** | Уже установлен; авто-offload; OpenAI-API; простота | Меньше контроля над параметрами |
| B | llama.cpp (server) | Максимальный контроль; тонкая настройка | Сложнее; нет prod API; ручное управление |
| C | vLLM | Production-grade; batched inference | Нет нативной Windows; избыточен для 1 GPU |

### Решение
**Ollama** (вариант A).

### Обоснование
Уже установлен и работает с RTX 5070 (qwen3:8b загружен, compute 12.0).
REST API совместим с OpenAI-форматом. Авто-GPU-offload критичен для
моделей >12 ГБ (30B MoE). Активное развитие (v0.32.0). Резерв — llama.cpp
для экспериментов с параметрами. Уверенность: **высокая**.

## TDR-002: API Gateway

### Контекст
Ollama API не имеет аутентификации. Безопасность — приоритет #1.
Нужен: auth, rate limiting, логирование, unified API, routing по моделям.

### Варианты

| # | Вариант | Плюсы | Минусы |
|---|---------|-------|--------|
| A | **LiteLLM Proxy** | API keys; rate limit; logging; routing; OpenAI-compat | Доп. Python-зависимость; +5-15 мс |
| B | Прямой Ollama API | Просто; нет overhead | Нет auth; нет logging; небезопасно |
| C | Open WebUI (как proxy) | UI + API | Фокус на UI, не на API-управлении |

### Решение
**LiteLLM Proxy** (вариант A).

### Обоснование
Безопасность — приоритет #1. API-ключи и логирование критичны.
Routing позволяет переключать модели по сценариям (chat-fast, coder-quality).
OpenAI-совместимый API обеспечивает совместимость с MCP-клиентами.
Доп. задержка 5-15 мс незначительна. Уверенность: **высокая**.

## TDR-003: Python Environment Manager

### Контекст
Python 3.10 установлен, но нет менеджера окружений с lock-файлами.
Воспроизводимость — приоритет #2. Нужен: venv + pip + lock-файлы.

### Варианты

| # | Вариант | Плюсы | Минусы |
|---|---------|-------|--------|
| A | **uv (Astral)** | Быстрее pip в 10-100x; lock-файлы; единый инструмент | Новый (2024); меньше docs |
| B | conda/miniconda | Научные пакеты; бинарные сборки | Тяжёлый; медленный; конфликты |
| C | venv + pip | Встроенный; стандарт | Медленный; нет lock-файлов |

### Решение
**uv** (вариант A).

### Обоснование
Воспроизводимость — приоритет #2. Lock-файлы обеспечивают точное
воспроизведение окружения. uv в 10-100x быстрее pip. Единый инструмент
заменяет venv + pip + pip-tools. Rust-based, активно развивается.
Уверенность: **высокая**.

## TDR-004: Secret Management

### Контекст
Секреты (API keys, конфиги) нужно хранить в git для воспроизводимости,
но в зашифрованном виде. Локальная станция, без KMS.

### Варианты

| # | Вариант | Плюсы | Минусы |
|---|---------|-------|--------|
| A | **SOPS + age** | Шифрование в git; без KMS; industry standard | Требует настройки keypair |
| B | .env + .gitignore | Просто | Секреты вне git; низкая воспроизводимость |
| C | HashiCorp Vault | Полное управление секретами | Избыточен для 1 машины; тяжёлый |

### Решение
**SOPS + age** (вариант A).

### Обоснование
Воспроизводимость + безопасность. age не требует KMS (локальный keypair).
Секреты в git (зашифрованы) — воспроизводимость при восстановлении.
SOPS — industry standard от Mozilla. Vault избыточен для локальной станции.
Уверенность: **высокая**.

Docker Desktop 29.2.1 уже установлен. WSL2 backend. Нужен для
DevContainers, Qdrant, Open WebUI.

### Варианты

бесплатен для личного использования). Резерв — Rancher Desktop
при лицензионных проблемах. Уверенность: **высокая**.

### Обоснование
Production-grade, Docker-образ, REST+gRPC API, фильтрация metadata.
Активно развивается. Интеграция с LangChain. Резерв — ChromaDB
для прототипов. Уверенность: **высокая**.

## TDR-008: RAG Framework

### Контекст
Нужен фреймворк для RAG: chunking, embedding, retrieval, augmentation.
Интеграция с Ollama, Qdrant, агентными возможностями.

### Варианты

| # | Вариант | Плюсы | Минусы |
|---|---------|-------|--------|
| A | **LangChain** | Широкая экосистема; интеграции; агентность | Сложный API; breaking changes |
| B | LlamaIndex | Фокус на RAG; лучшая индексация | Меньше экосистема; меньше агентности |
| C | Кастомный pipeline | Полный контроль; нет зависимостей | Время разработки; ошибки |

### Решение
**LangChain** (вариант A).

### Обоснование
Широкая экосистема, интеграции с Qdrant и Ollama из коробки.
Агентные возможности (вместе с LangGraph). Резерв — LlamaIndex
для специализированных RAG-задач. Уверенность: **высокая**.

## TDR-011: Monitoring

### Контекст
Мониторинг VRAM, RAM, статуса сервисов. Локальная станция, 1 машина.
Безопасность и надёжность — приоритеты.

### Варианты

| # | Вариант | Плюсы | Минусы |
|---|---------|-------|--------|
| A | **PowerShell-скрипты + nvidia-smi** | Нулевые зависимости; быстро; настраиваемо | Нет исторических данных; нет дашбордов |
| B | Prometheus + Grafana | Production-grade; метрики; дашборды; алерты | Тяжёлый; overhead; избыточен |
| C | Ollama built-in stats | Просто | Ограничено; только Ollama |

### Решение
**PowerShell-скрипты** (вариант A).

### Обоснование
Для 1 машины Prometheus+Grafana избыточен. PowerShell-скрипты
обеспечивают health checks, VRAM monitoring, алерты без overhead.
Простота использования (приоритет #7). Резерв — Prometheus+Grafana
при потребности в исторических данных. Уверенность: **высокая**.

## TDR-012: Model Selection Strategy

### Контекст
12 ГБ VRAM ограничивает размер моделей. Качество в приоритете над
скоростью. Нужны профили для разных сценариев: Chat, Coding, Vision,
Embeddings. Поддержка tool calling для агентности.

### Варианты стратегии

| # | Стратегия | Плюсы | Минусы |
|---|-----------|-------|--------|
| A | **Профили Low/Medium/High VRAM** | Гибкость; модель под сценарий; качество vs скорость | Сложнее управление; больше моделей |
| B | Одна модель для всего | Просто; одна модель в VRAM | Компромисс по качеству/скорости |
| C | Только максимальные модели | Максимальное качество | Медленно; VRAM конфликт; не для интерактива |

### Решение
**Профили Low/Medium/High VRAM** (вариант A).

### Обоснование
Качество в приоритете, но интерактивные сценарии (code completion)
требуют низкой задержки. Профили дают гибкость: 7B для быстрых задач,
30B MoE для сложных. 64 ГБ DDR5 обеспечивает CPU offload для крупных
моделей. 10 моделей покрывают все сценарии. Уверенность: **высокая**.

### Выбранные модели

| Профиль | Chat | Coding | Vision | Embeddings |
|---------|------|--------|--------|------------|
| Low VRAM | qwen3:8b | qwen2.5-coder:7b | qwen2.5vl:3b | nomic-embed-text |
| Medium VRAM | qwen2.5:14b | deepseek-coder-v2:lite | qwen2.5vl:7b | bge-m3 |
| High VRAM | qwen2.5:32b | qwen3-coder:30b-a3b | qwen2.5vl:32b | bge-m3 |

через локальные модели с tool calling.

| # | Вариант | Плюсы | Минусы |
|---|---------|-------|--------|
| A | **Continue** | MCP-совместимый; VS Code; Ollama support | Меньше функций чем Koda |
