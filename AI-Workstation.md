# AI Workstation — спецификация компонентов

> Основание: Requirements.md (подтверждён пользователем).
> Принципы: исключительно локально, качество приоритетнее скорости,
> модели обновляются, vision в baseline, без ограничений по времени/объёму.

---

## 1. Компоненты системы

### 1.1. Inference Engine

| Параметр | Основной вариант | Резервный вариант |
|----------|------------------|-------------------|
| **Название** | Ollama | llama.cpp (server mode) |
| **Назначение** | Локальный inference LLM: загрузка GGUF-моделей, GPU/CPU offload, REST API | Прямой inference через llama.cpp с максимальным контролем параметров |
| **Преимущества** | Уже установлен и работает; авто-GPU-offload; OpenAI-совместимый API; простота управления моделями; активное развитие | Тонкая настройка quantization, batch size, KV cache; прямой доступ к низкоуровневым параметрам |
| **Недостатки** | Меньше контроля над параметрами inference; ограниченная настройка KV cache | Сложнее управлять; нет встроенного API-сервера prod-уровня; ручное управление моделями |
| **Требования** | NVIDIA driver 591.86+, 12 ГБ VRAM, 64 ГБ RAM | То же + ручная сборка/настройка |
| **Стратегия обновления** | `ollama --version` проверка; автообновление через скрипт обслуживания | Ручная сборка из source при необходимости |
| **Причины замены** | Ollama перестанет поддерживать Blackwell / GGUF; критическая уязвимость без патча | Ollama не удовлетворяет потребности в тонкой настройке |

**Решение:** Ollama (основной). Уже работает с RTX 5070 (compute 12.0),
qwen3:8b загружен. Резерв — llama.cpp для экспериментов с параметрами.

### 1.2. API Gateway / Proxy

| Параметр | Основной вариант | Резервный вариант |
|----------|------------------|-------------------|
| **Название** | LiteLLM Proxy | Прямой Ollama API (localhost:11434) |
| **Назначение** | Unified OpenAI-совместимый API-шлюз: аутентификация, rate limiting, логирование, маршрутизация запросов к моделям | Прямой доступ к Ollama без промежуточного слоя |
| **Преимущества** | API-ключи (безопасность); rate limiting; логирование всех запросов; unified API для всех моделей; routing по сценариям | Максимальная простота; нет доп. зависимости; минимальная задержка |
| **Недостатки** | Доп. Python-зависимость; +5-15 мс задержки; требует настройки | Нет auth; нет rate limiting; нет логирования; небезопасно при доступе из сети |
| **Требования** | Python 3.10+, pip/uv, Ollama запущен | Ollama запущен |
| **Стратегия обновления** | `pip install --upgrade litellm` через uv | — |
| **Причины замены** | LiteLLM заброшен; критическая уязвимость; избыточность при локальном использовании без сети | Потребность в auth/logging/rate-limiting |

**Решение:** LiteLLM Proxy (основной). Безопасность — приоритет #1.
API-ключи и логирование критичны. Резерв — прямой API для отладки.

### 1.3. Python Environment Manager

| Параметр | Основной вариант | Резервный вариант |
|----------|------------------|-------------------|
| **Название** | uv (Astral) | venv + pip (стандартные) |
| **Назначение** | Управление Python-окружениями: venv, pip, lock-файлы, версии Python | Стандартное виртуальное окружение + pip |
| **Преимущества** | На порядки быстрее pip; единый инструмент (venv+pip+pip-tools); lock-файлы для воспроизводимости; Rust-based | Встроен в Python; стандарт; нет доп. зависимостей |
| **Недостатки** | Новый (2024); меньше документации в экосистеме | Медленный; нет lock-файлов; ручное управление зависимостями |
| **Требования** | Установка через pip или PowerShell | Python 3.10+ (уже есть) |
| **Стратегия обновления** | `uv self update` | Обновление pip: `python -m pip install --upgrade pip` |
| **Причины замены** | uv заброшен; несовместимость с критичным пакетом | — (всегда доступен как fallback) |

**Решение:** uv (основной). Воспроизводимость — приоритет #2.
Lock-файлы обеспечивают точное воспроизведение окружения.

| **Стратегия обновления** | `sops --version` проверка; обновление через package manager | — |

**Решение:** SOPS + age (основной). Воспроизводимость + безопасность.
Резерв — .env для локальной разработки без коммита.

## 2. Модельные профили

Профили определяют стратегию выбора моделей в зависимости от
доступной VRAM и приоритета (качество > скорость).

### 2.1. Профиль Low VRAM (быстрый интерактив)

> ~4-6 ГБ VRAM доступно (модель + контекст).
> Сценарий: code completion, быстрый чат, когда VRAM занят другими процессами.

| Тип | Модель | Размер (Q4_K_M) | Контекст | VRAM | Назначение |
|-----|--------|-----------------|----------|------|------------|
| **Chat** | qwen3:8b | 5.2 ГБ | 40K | ~6 ГБ | Универсальный чат, thinking mode |
| **Coding** | qwen2.5-coder:7b | 4.7 ГБ | 32K | ~5.5 ГБ | Code completion, рефакторинг |
| **Vision** | qwen2.5vl:3b | 3.2 ГБ | 125K | ~4 ГБ | Базовый анализ изображений |
| **Embeddings** | nomic-embed-text | 274 МБ | 8K | ~0.3 ГБ | Embeddings для RAG |

### 2.2. Профиль Medium VRAM (сбалансированный)

> ~8-10 ГБ VRAM доступно (модель + контекст).
> Сценарий: основная дневная работа, качественный кодинг, RAG.

| Тип | Модель | Размер (Q4_K_M) | Контекст | VRAM | Назначение |
|-----|--------|-----------------|----------|------|------------|
| **Chat** | qwen2.5:14b | ~9 ГБ | 32K | ~10 ГБ | Качественный чат, reasoning |
| **Coding** | deepseek-coder-v2:lite | 8.9 ГБ | 160K | ~10 ГБ | MoE, качественный кодинг, длинный контекст |
| **Vision** | qwen2.5vl:7b | 6.0 ГБ | 125K | ~7 ГБ | Качественный анализ изображений, OCR |
| **Embeddings** | bge-m3 | ~1.2 ГБ | 8K | ~1.5 ГБ | Мультиязычные embeddings, высокая точность |

### 2.3. Профиль High VRAM (максимальное качество)

> 12 ГБ VRAM + CPU offload (64 ГБ DDR5).
> Сценарий: сложные агентные задачи, рефакторинг крупных репозиториев,
> максимальное качество. Допускается медленный inference.

| Тип | Модель | Размер (Q4_K_M) | Контекст | VRAM + RAM | Назначение |
|-----|--------|-----------------|----------|------------|------------|
| **Chat** | qwen2.5:32b | ~20 ГБ | 32K | 12 ГБ VRAM + ~8 ГБ RAM | Максимальное качество чата, reasoning |
| **Coding** | qwen3-coder:30b-a3b | 19 ГБ | 256K | 12 ГБ VRAM + ~7 ГБ RAM | MoE (3B active), максимальное качество кода, 256K контекст |
| **Vision** | qwen2.5vl:32b | 21 ГБ | 125K | 12 ГБ VRAM + ~9 ГБ RAM | Максимальное качество vision, анализ сложных диаграмм |
| **Embeddings** | bge-m3 | ~1.2 ГБ | 8K | ~1.5 ГБ VRAM | Тот же — embeddings не требуют масштабирования |

### 2.6. Стратегия обновления моделей

| Действие | Команда | Частота |
|----------|---------|---------|
| Проверка новых моделей | `ollama list` + мониторинг ollama.com/library | При выходе новых версий |
| Обновление модели | `ollama pull <model>` (обновляет до latest) | По необходимости |
| Смена модели | `ollama pull <new>; ollama rm <old>` | По необходимости |
| Проверка digest | `ollama show <model> --modelfile` | При обновлении |
| Очистка неиспользуемых | `ollama list` → `ollama rm <unused>` | Еженедельно |
