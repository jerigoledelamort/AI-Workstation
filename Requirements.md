## 1. Аппаратная платформа

### 1.1. Операционная система

| Параметр | Значение |
|----------|----------|
| ОС | Microsoft Windows 11 Home |
| Сборка | 26200 |
| Разрядность | 64-разрядная |
| Файловая система | NTFS |

### 1.2. Центральный процессор

| Параметр | Значение |
|----------|----------|
| Модель | AMD Ryzen 7 7800X3D |
| Ядра / Потоки | 8 / 8 |
| Базовая частота | 4.2 GHz |
| Кэш L3 | 96 MB (3D V-Cache) |
| Архитектура | Zen 4 (Raphael) |

**Следствие для проектирования:** 3D V-Cache даёт преимущество для
инференса моделей, не помещающихся в VRAM (CPU offload). 8 потоков
ограничивают параллелизм — не более 1-2 одновременных inference-запросов
при CPU-offload.

### 1.3. Оперативная память

| Параметр | Значение |
|----------|----------|
| Всего | 64 ГБ |
| Свободно | ~48 ГБ |
| Тип | DDR5 |

**Следствие:** 64 ГБ RAM достаточно для:
- MoE-моделей до ~40 ГБ (Q4_K_M) с частичным CPU offload;
- RAG-пайплайнов с векторными БД в оперативной памяти;
- одновременной работы Docker, WSL2, Ollama и IDE.

### 1.4. Графический процессор

| Параметр | Значение |
|----------|----------|
| Модель | NVIDIA GeForce RTX 5070 |
| VRAM | 12 227 МБ (~11.9 ГБ) |
| Свободно | ~9 774 МБ (~9.5 ГБ) |
| Compute Capability | 12.0 (Blackwell) |
| Driver | 591.86 (32.0.15.9186, от 2026-01-20) |

**Следствие для проектирования:** 12 ГБ VRAM — ключевой ограничивающий
фактор. Полностью в VRAM помещаются:
- модели 7B-8B (Q4_K_M): ~4.5-5.5 ГБ + контекст;
- модели до ~10 ГБ с Q4 при ограниченном контекстном окне.

Модели 14B+ требуют CPU offload, что снижает скорость.

### 1.5. Хранилище

| Диск | Тип | Объём | Свободно | Интерфейс |
|------|-----|-------|----------|-----------|
| C: | Samsung 990 PRO 2TB | 1861.5 ГБ | 1558.1 ГБ | NVMe |
| D: | Seagate ST4000 4TB | 3726 ГБ | 1550.2 ГБ | SATA HDD |

**Следствие:** NVMe (C:) — для ОС, инструментов, моделей, Docker-образов.
HDD (D:) — для архивов, бэкапов, крупных датасетов. Модели Ollama
(~5-20 ГБ каждая) должны храниться на NVMe для скорости загрузки в VRAM.

### 1.6. CUDA и драйверы

| Компонент | Версия |
|-----------|--------|
| NVIDIA Driver | 591.86 |
| CUDA Toolkit (nvcc) | 12.8 (release V12.8.61) |
| PyTorch CUDA | 11.8 (torch 2.7.1+cu118) |

**⚠️ Несоответствие:** CUDA Toolkit 12.8 установлен, но PyTorch собран
под CUDA 11.8. Это работает (обратная совместимость драйвера), но
ограничивает возможности PyTorch. Для RTX 5070 (Blackwell, compute 12.0)
рекомендуется PyTorch с поддержкой CUDA 12.x при необходимости прямого
GPU-инференса через PyTorch. Ollama использует собственный runtime
(не зависит от PyTorch) и работает корректно.

### 1.7. Сеть

| Параметр | Значение |
|----------|----------|
| Download | ~105 Mbps |
| Upload | ~26 Mbps |

**Следствие:** загрузка моделей из Ollama-репозитория:
- 7B модель (~4.7 ГБ): ~6 минут;
- 14B модель (~9 ГБ): ~11 минут;
- 30B MoE модель (~19 ГБ): ~24 минуты.

### 1.8. Допустимый объём загрузок и время установки

| Параметр | Оценка |
|----------|--------|
| Доступно на NVMe (C:) | ~1558 ГБ |
| Доступно на HDD (D:) | ~1550 ГБ |
| Прогноз загрузок для baseline-конфигурации | ~15-25 ГБ (модели + Docker-образы + инструменты) |
| Прогноз загрузок для расширенной конфигурации | ~40-60 ГБ (доп. модели, векторные БД, датасеты) |
| Допустимое время установки | Требует подтверждения пользователя. Оценка: 1-3 часа для baseline, 3-5 часов для расширенной |

## 3. Сценарии использования

Определены по контексту: 6 проектов в `D:\Projects`, установленные
AI-ассистенты (Koda, GigaCode), Ollama с qwen3:8b, PyTorch с CUDA.

### 3.1. Программирование с AI-ассистентом (основной сценарий)

- **Описание:** ежедневная работа с VS Code + Koda/GigaCode, дополненный
  локальными моделями через Ollama для code completion, рефакторинга,
  отладки.
- **Требования:** низкая задержка (<500 мс first token), контекст >=32K
  токенов, поддержка tool calling для агентных операций.
- **Модели:** qwen2.5-coder:7b (primary), qwen3:8b (fallback universal).
- **Частота:** постоянная, ежедневно.

### 3.2. Работа с большими репозиториями

- **Описание:** анализ и навигация по крупным кодовым базам (проекты
  AstraGlyph, deus-ex-machina, isaac-atlas и др.).
- **Требования:** контекстное окно >=32K, возможность RAG по коду
  (embeddings + векторный поиск).
- **Ограничение:** 12 ГБ VRAM ограничивает размер модели + контекста.
  Для больших репозиториев — RAG предпочтительнее длинного контекста.

### 3.3. Написание документации

- **Описание:** генерация и редактирование технической документации,
  README, архитектурных документов.
- **Требования:** качественная генерация текста, понимание Markdown/AsciiDoc.
- **Модели:** qwen3:8b (универсальная, с thinking).

### 3.4. Локальный чат (interactive)

- **Описание:** интерактивный диалог с локальной моделью для брейншторма,
  прототипирования, вопросов.
- **Требования:** streaming, низкая задержка, удобный UI (Open WebUI или
  аналогичный).
- **Модели:** qwen3:8b.

### 3.5. RAG (Retrieval-Augmented Generation)

- **Описание:** поиск и генерация ответов по локальным документам,
  кодовым базам, датасетам.
- **Требования:**
  - embedding-модель (локальная, через Ollama);
  - векторная БД (Qdrant/Chroma);
  - pipeline: chunk -> embed -> store -> retrieve -> augment -> generate.
- **Модели:** embedding (nomic-embed-text или bge-m3), генерация (qwen3:8b).

### 3.6. MCP (Model Context Protocol)

- **Описание:** интеграция локальных моделей с MCP-серверами для
  доступа к файлам, Git, терминалу, веб-поиску из AI-ассистента.
- **Требования:** Ollama с tool calling, MCP-совместимый клиент
  (VS Code + расширение, Continue, или аналогичный).
- **Ограничение:** не все локальные модели поддерживают tool calling
  корректно. qwen3:8b поддерживает (capabilities: tools).

### 3.7. Агентная работа

- **Описание:** многошаговые автономные задачи — анализ кодовой базы,
  рефакторинг, генерация тестов, создание PR.
- **Требования:** модель с сильным reasoning + tool calling,
  агентный фреймворк (LangGraph / CrewAI / аналогичный),
  изоляция среды выполнения (DevContainers).
- **Модели:** qwen3:8b (thinking mode), опционально qwen3-coder:30b-a3b
  (MoE, с CPU offload, медленнее).
- **Риск:** агентные задачи требуют много итераций -> высокое потребление
  VRAM/RAM -> нужна стратегия освобождения ресурсов.

### 3.8. Анализ изображений (опционально)

- **Описание:** OCR, анализ скриншотов, диаграмм, UI-мокапов.
- **Требования:** мультимодальная модель (vision-language).
- **Ограничение:** 12 ГБ VRAM. Мультимодальные модели (LLaVA, Qwen2-VL)
  требуют дополнительной VRAM для vision encoder.
- **Модели:** llava:7b или qwen2.5-vl:7b (если доступны в Ollama).
- **Приоритет:** baseline (подтверждено пользователем).

## 4. Приоритеты пользователя

Определены по контексту задачи (локальная AI Workstation, без облачных
зависимостей) и требуют подтверждения.

### 4.1. Предполагаемые приоритеты (от высокого к низкому)

| # | Приоритет | Обоснование |
|---|-----------|-------------|
| 1 | **Максимальная приватность** | Локальная станция, без передачи данных вовне. Все модели и данные остаются на машине. |
| 2 | **Полностью автономная работа** | Без зависимости от внешних API (OpenAI, Anthropic). Интернет нужен только для загрузки моделей/инструментов. |
| 3 | **Надёжность** | Воспроизводимые окружения, health checks, автоматическое восстановление. |
| 4 | **Сопровождаемость** | Декларативная конфигурация, документация, автоматизация обслуживания. |
| 5 | **Производительность** | Вторична по отношению к качеству. Низкая задержка важна только для интерактивных сценариев (чат, code completion). |
| 6 | **Масштабируемость** | Возможность добавлять модели, инструменты, проекты без перестройки архитектуры. |
| 3 | **Максимальное качество моделей** | Качество в приоритете над скоростью (подтверждено). Использование лучших доступных моделей, включая 30B MoE с CPU offload. |
| 8 | **Минимальное энергопотребление** | Не критично (подтверждено пользователем). Энергосбережение не настраивается. |

### 4.2. Архитектурные приоритеты (из исходной задачи)

|---|-----------|--------|
| 1 | Безопасность | Задан |
| 2 | Воспроизводимость | Задан |
| 3 | Надёжность | Задан |
| 4 | Сопровождаемость | Задан |
| 5 | Масштабируемость | Задан |

### 4.3. Принятые решения

1. **Приватность vs. качество:** ✅ Исключительно локально. Облачные API
   не используются — это фундаментальный принцип станции. Качество
   ограничено возможностями локальных моделей в рамках 12 ГБ VRAM.

2. **Автономность vs. актуальность:** ✅ Обновление и смена моделей
   предусматриваются. Станция не полностью офлайн — интернет
   используется для загрузки новых моделей, обновления инструментов
   и инструментов. Модели не фиксируются, стратегия обновления —
   часть автоматизации.

3. **Производительность vs. качество:** ✅ Качество в приоритете.
   Допускается использование более крупных моделей (30B MoE) с CPU
   offload, даже если это медленнее. 7B модели — для интерактивных
   сценариев с низкой задержкой, 30B+ — для сложных задач, где
   качество критично.

4. **Энергопотребление:** ✅ Не критично. Режим энергосбережения
   не настраивается. GPU и CPU работают на максимальной мощности.

5. **Анализ изображений:** ✅ Да. Vision-модуль включается в baseline.
   Требуется мультимодальная модель (vision-language) в Ollama.

6. **Допустимое время установки:** ✅ Не критично. Полная расширенная
   конфигурация без ограничений по времени.

7. **Допустимый объём загрузок:** ✅ Не критично. Доступно ~1558 ГБ
   на NVMe + ~1550 ГБ на HDD. Ограничений нет.

## 5. Ограничения и риски

### 5.1. Аппаратные ограничения

| Ограничение | Влияние | Митигация |
|-------------|---------|-----------|
| 12 ГБ VRAM | Модели >10 ГБ требуют CPU offload | Стратегия: 7B в VRAM, 30B MoE — по требованию |
| 8 потоков CPU | Ограниченный параллелизм при offload | Не более 1-2 одновременных inference-запросов |
| HDD на D: | Медленный случайный доступ | Модели и Docker — только на NVMe (C:) |
| Один GPU | Конкуренция за VRAM между процессами | Разделять во времени, не запускать PyTorch и Ollama одновременно |

### 5.2. Программные риски

| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| Несовместимость PyTorch CUDA 11.8 с Blackwell | Средняя | Ограничение PyTorch-инференса | Обновить PyTorch до cu12.x при необходимости |
| Docker Desktop лицензионные ограничения | Низкая | Для коммерческого использования | Оценить альтернативы (Rancher Desktop, WSL2 + Docker CE) |
| Ollama API без аутентификации | Высокая | Несанкционированный доступ из локальной сети | API-прокси (LiteLLM) с auth, firewall правила |
| Конкуренция VRAM: Koda/GigaCode + Ollama | Средняя | OOM, замедление | Мониторинг VRAM, стратегия выгрузки моделей |

### 5.3. Ограничения воспроизводимости

- Windows 11 Home — нет Group Policy, ограниченные средства управления
  безопасностью по сравнению с Pro/Enterprise.
- Docker Desktop на WSL2 — слой виртуализации, отличия от нативного Linux.
- Python 3.10 — не последний (3.12/3.13 актуальнее), но достаточно
  стабилен для большинства AI-инструментов.
